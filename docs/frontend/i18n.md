# Frontend i18n (next-intl + ICU)

> Reference for build-fe SubAgent and frontend-implementer.
> ⚙️ **Optional integration** — activated via init's "Internationalization" question (`manifest.optional_integrations.i18n=true`). Until activated, this document is reference-only; setup steps live in `recipes/i18n-setup.md`.
> next-intl with ICU MessageFormat for webstack's Next.js 16 App Router with FSD-lite layout.

## What is webstack i18n

webstack uses **next-intl** (4.X) as its first-pick i18n library. It integrates directly with the Next.js 16 App Router — Server Components, Client Components, Server Actions, and middleware are all first-class citizens.

Key capabilities:

- **Server-first.** `getTranslations()` runs in any Server Component or Action; messages load on the server and never inflate the client bundle.
- **ICU MessageFormat.** Pluralization, gender/select, number/date formatting — all natively supported. Grammar rules stay in JSON.

## Why next-intl

next-intl mirrors the RSC split with two APIs on the same namespace:

- `getTranslations('Namespace')` — async, Server Components and Server Actions.
- `useTranslations('Namespace')` — hook, Client Components (`'use client'`).

Both read from `messages/<locale>.json`. `createNavigation()` provides locale-aware `<Link>`, `useRouter`, `usePathname`, and `redirect` wrappers that automatically prefix URLs with the active locale.

## webstack convention

Global messages live at `messages/<locale>.json` (project root). Feature-specific messages live inside the FSD slice — see [§ FSD message location convention](#fsd-message-location-convention).

### Configuration files

```
src/i18n/
├── routing.ts      # defineRouting — locales, defaultLocale
├── request.ts      # getRequestConfig — locale detection per request
└── navigation.ts   # createNavigation — Link, useRouter, redirect
src/app/[locale]/
├── layout.tsx      # locale layout — validates, sets locale, provides messages
└── page.tsx
middleware.ts       # project root — createMiddleware(routing)
```

**`src/i18n/routing.ts`** — single source of truth for all locale config:

```ts
import { defineRouting } from 'next-intl/routing'
export const routing = defineRouting({ locales: ['en', 'ko', 'ja'], defaultLocale: 'en' })
```

**`src/i18n/request.ts`** — per-request locale resolution:

```ts
import { getRequestConfig } from 'next-intl/server'
import { hasLocale } from 'next-intl'
import { routing } from './routing'

export default getRequestConfig(async ({ requestLocale }) => {
  const requested = await requestLocale
  const locale = hasLocale(routing.locales, requested) ? requested : routing.defaultLocale
  return { locale, messages: (await import(`../../messages/${locale}.json`)).default }
})
```

**`middleware.ts`** (project root) — locale detection and redirect:

```ts
import createMiddleware from 'next-intl/middleware'
import { routing } from './src/i18n/routing'
export default createMiddleware(routing)
export const config = { matcher: '/((?!api|trpc|_next|_vercel|.*\\..*).*)'  }
```

**`next.config.ts`** — wrap with the plugin so `request.ts` is registered:

```ts
import createNextIntlPlugin from 'next-intl/plugin'
export default createNextIntlPlugin()(/* your NextConfig */)
```

**`src/i18n/navigation.ts`** — locale-aware navigation primitives:

```ts
import { createNavigation } from 'next-intl/navigation'
import { routing } from './routing'
export const { Link, redirect, usePathname, useRouter } = createNavigation(routing)
```

Import `Link`/`useRouter` from `@/i18n/navigation`, never from `next/link` or `next/navigation`, so URLs are always locale-prefixed.

## Locale routing

webstack uses **path-prefix routing** — the locale is the first URL segment (`/en/dashboard`, `/ko/dashboard`). Subdomain routing (`ko.example.com`) is supported but not the webstack default; it requires DNS wildcard configuration.

**`src/app/[locale]/layout.tsx`:**

```tsx
import { NextIntlClientProvider, hasLocale } from 'next-intl'
import { notFound } from 'next/navigation'
import { setRequestLocale, getMessages } from 'next-intl/server'
import { routing } from '@/i18n/routing'

export const generateStaticParams = () =>
  routing.locales.map((locale) => ({ locale }))

export default async function LocaleLayout({ children, params }) {
  const { locale } = await params
  if (!hasLocale(routing.locales, locale)) notFound()
  setRequestLocale(locale)
  return (
    <html lang={locale}>
      <body>
        <NextIntlClientProvider messages={await getMessages()}>
          {children}
        </NextIntlClientProvider>
      </body>
    </html>
  )
}
```

`generateStaticParams` pre-renders all locale variants at build time. `setRequestLocale` must be called in every layout and page so next-intl can determine the locale in static context.

## ICU MessageFormat

next-intl supports ICU MessageFormat natively — no extra packages. ICU keeps complex grammar rules in JSON and out of application code.

### Pluralization

```json
{ "Cart": { "itemCount": "{count, plural, =0 {No items} one {# item} other {# items}}" } }
```

```tsx
const t = await getTranslations('Cart')
t('itemCount', { count: 3 })  // → "3 items"
```

`#` expands to the formatted count. `=0` matches exact zero before `one`/`other` keywords.

### Gender and select

```json
{ "Invite": { "sentMessage": "{gender, select, female {She invited you} male {He invited you} other {They invited you}}" } }
```

```tsx
t('sentMessage', { gender: user.gender })
```

`select` works for any fixed-choice enumeration. `other` is mandatory and acts as the fallback.

### Number and date formatting

Use `getFormatter` (server) / `useFormatter` (client) for imperative locale-aware formatting, or ICU skeletons (`::`) inside message strings:

```ts
const format = await getFormatter()
format.number(1234567.89, { style: 'currency', currency: 'USD' })
// en → "$1,234,567.89"  ko → "US$1,234,567.89"
format.dateTime(new Date(), { dateStyle: 'long' })
// en → "May 3, 2026"  ko → "2026년 5월 3일"
```

```json
{ "Invoice": { "total": "Total: {amount, number, ::currency/USD}" } }
```

The `::` prefix marks an ICU skeleton — portable and locale-safe.

### Nested patterns

Place `select` outside `plural` when combining both, as per ICU best practices:

```json
{
  "Notification": {
    "newMessages": "{gender, select, female {She has {count, plural, one {# message} other {# messages}}} other {They have {count, plural, one {# message} other {# messages}}}}"
  }
}
```

## RTL support

### `dir` on `<html>`

Derive `dir` from the locale in the locale layout:

```tsx
const RTL_LOCALES = new Set(['ar', 'he', 'fa'])

export default async function LocaleLayout({ children, params }) {
  const { locale } = await params
  const dir = RTL_LOCALES.has(locale) ? 'rtl' : 'ltr'
  return <html lang={locale} dir={dir}>...</html>
}
```

Setting `dir` on `<html>` is sufficient — browser UA stylesheets and CSS logical properties respect it automatically.

### Tailwind logical properties

Tailwind CSS v4 ships logical property utilities in core. Use them in all components so layouts mirror in RTL without JavaScript direction checks:

| Avoid (physical) | Use (logical) |
|---|---|
| `ml-*` / `mr-*` | `ms-*` / `me-*` (margin-inline-start/end) |
| `pl-*` / `pr-*` | `ps-*` / `pe-*` (padding-inline-start/end) |
| `text-left` / `text-right` | `text-start` / `text-end` |
| `border-l-*` / `border-r-*` | `border-s-*` / `border-e-*` |

```tsx
<div className="ms-4 ps-2 text-start">...</div>  {/* LTR: left  RTL: right */}
```

## Lazy messages

By default, `getRequestConfig` loads the entire locale file on every request. For large apps, split per feature: each feature's layout segment loads only its own file.

```ts
// src/app/[locale]/(checkout)/layout.tsx
export default async function CheckoutLayout({ children, params }) {
  const { locale } = await params
  const messages = (await import(`@/features/checkout/messages.${locale}.json`)).default
  return <NextIntlClientProvider messages={messages}>{children}</NextIntlClientProvider>
}
```

Default recommendation: keep all messages in `messages/<locale>.json`. In RSC, unused keys never reach the browser — they are inlined at render time, not hydrated as JS.

## FSD message location convention

Messages live at two levels:

**Global** (`messages/<locale>.json` at project root) — shared strings used by multiple slices: navigation, error messages, common UI labels.

**Feature-scoped** (`src/features/<feature>/messages.<locale>.json`) — strings owned exclusively by one slice. Co-locate with the slice; load lazily via the feature's layout segment (see [§ Lazy messages](#lazy-messages)).

```
src/features/checkout/
├── ui/CheckoutForm.tsx
├── messages.en.json   ← namespace key: "Checkout"
├── messages.ko.json
└── index.ts
```

Use the PascalCase slice name as the top-level namespace key (`"Checkout": { ... }`) to avoid collisions with global messages.

## Anti-patterns

**Hardcoded strings in JSX.** Every rendered string must come from a `t()` call. Hardcoded English text is untranslatable.

**Key-less translations.** Using the full sentence as the key (`t('Hello, world!')`) — any edit to English breaks all translations. Use semantic dot-notated keys (`t('HomePage.greeting')`).

**No fallback locale.** Always set `defaultLocale` in `defineRouting` and guard with `hasLocale` in `getRequestConfig`. Without a fallback, an unsupported locale segment throws instead of returning 404.

**Manual concatenation instead of ICU.** Concatenating translated fragments breaks word order in Korean, Arabic, and others.

```ts
// Wrong — word order breaks
t('prefix') + userName + t('suffix')

// Correct — ICU placeholder, locale controls placement
t('welcomeMessage', { name: userName })
// "Welcome, {name}!" / "{name}님 환영합니다!"
```

**Using `next/link` directly after enabling i18n routing.** Import `Link` and `useRouter` from `@/i18n/navigation` — the next/link version never prefixes the locale.

**Forgetting `setRequestLocale` in static pages.** next-intl throws at build time if it cannot determine the locale in static context. Call it in every page and layout.

**No `dir` on `<html>`.** RTL locales without `dir` produce mirrored layouts. Set it dynamically in the locale layout.
